# Modal MCP

Self-hosted Model Context Protocol server for read-only Modal operations.

The v1 server exposes Modal discovery, app, deployment, log, container, volume,
and sandbox read APIs over FastMCP Streamable HTTP at `/mcp`. Mutating `change`
and `expert` toolsets are registered as disabled stubs, hidden from default
`tools/list`, and guarded by policy code for future releases.

## Quickstart

Install dependencies, generate local server settings, add dedicated Modal token
files, and verify the setup before starting the server:

```bash
uv sync --extra dev
modal-mcp setup --yes
mkdir -p .secrets
printf '%s' '<modal-token-id>'     > .secrets/modal-token-id
printf '%s' '<modal-token-secret>' > .secrets/modal-token-secret
chmod 600 .secrets/*
printf '\nMODAL_TOKEN_ID_FILE=%s/.secrets/modal-token-id\n' "$PWD" >> .env
printf 'MODAL_TOKEN_SECRET_FILE=%s/.secrets/modal-token-secret\n' "$PWD" >> .env
modal-mcp doctor --env-file .env
modal-mcp run --env-file .env
```

`setup --yes` writes two files (both mode `0600`) and exits; it is idempotent:

- `.env` — server config (signing key path + allowed origins; no credentials)
- `.secrets/signing-key.txt` — HMAC signing key

To start the server manually with raw environment variables instead:

```bash
export MODAL_MCP_ALLOWED_ORIGINS=http://127.0.0.1:8765
export MODAL_MCP_SIGNING_KEYS=kid1:$(openssl rand -hex 32)
uv run modal-mcp
```

## Credential Safety and Service-User Recommendation

**`modal-mcp setup` never reads or writes Modal credentials.**  The `.env` file
generated by `setup --yes` intentionally omits `MODAL_TOKEN_ID`,
`MODAL_TOKEN_SECRET`, and `MODAL_ENVIRONMENT`.  Modal credentials must be
supplied separately.

**Recommendation:** create a dedicated Modal service-user with Viewer
permissions scoped to a non-production workspace, then store its token as
file-backed secrets:

```bash
mkdir -p .secrets
printf '%s' '<modal-token-id>'     > .secrets/modal-token-id
printf '%s' '<modal-token-secret>' > .secrets/modal-token-secret
chmod 600 .secrets/*
```

Export the file paths at runtime, add them to your shell profile, or append
the file paths to the generated `.env`:

```bash
export MODAL_TOKEN_ID_FILE=$PWD/.secrets/modal-token-id
export MODAL_TOKEN_SECRET_FILE=$PWD/.secrets/modal-token-secret
```

`modal-mcp setup` does **not** automatically create a scoped service-user
token.  You must create the service-user and token yourself through the Modal
dashboard or CLI (`modal token new`).

If `~/.modal.toml` already exists, `modal-mcp doctor` will warn that those
credentials may belong to a personal or admin account with broader permissions
than the read-only server requires.

## Diagnostics

Run `modal-mcp doctor` after setup to verify the installation before starting
the server:

```bash
modal-mcp doctor
```

`doctor` checks (without loading full server settings):

1. Package imports — `modal_mcp`, `modal`, `fastmcp`, `uvicorn`
2. `.env` file presence
3. Signing key configuration
4. `MODAL_MCP_ALLOWED_ORIGINS` value
5. Read-only readiness (`MODAL_MCP_READ_ONLY`)
6. Enabled toolsets (`MODAL_MCP_ENABLED_TOOLSETS`)
7. Modal credential source (`~/.modal.toml`, env vars, or file-backed paths)
8. Modal SDK auth health (only when credentials are present)
9. Modal CLI presence

Exit code is `0` when all checks pass; `3` when warnings exist but no check
fails (partial-ready); `1` when at least one check fails.

## Agent Integration

Print the config block for your agent client, then paste it into the client's
config file:

```bash
# Codex CLI (stdio transport — Codex launches modal-mcp as a subprocess)
modal-mcp print-agent-config --target codex

# Claude Desktop (SSE transport — server must be started separately)
modal-mcp print-agent-config --target claude
```

To install automatically into Codex CLI or Claude Desktop:

```bash
# Codex — preview then write the [mcp_servers.modal-mcp] entry to ~/.codex/config.toml
modal-mcp setup --install codex --env-file /absolute/path/to/.env --dry-run
modal-mcp setup --install codex --env-file /absolute/path/to/.env --yes

# Claude Desktop — preview then write the mcpServers.modal-mcp entry to claude_desktop_config.json
modal-mcp setup --install claude --dry-run
modal-mcp setup --install claude --yes
```

`--install codex` backs up the existing config, writes atomically, validates
the round-trip, and is idempotent (a no-op when the entry is already correct).
`--install claude` works the same way but writes an SSE URL entry to
`claude_desktop_config.json`; it does not require `--env-file` because Claude
Desktop connects over SSE to a separately-started server.

## Configuration

Required for self-hosted startup:

- `MODAL_MCP_ALLOWED_ORIGINS`: comma-separated browser/client origins allowed to
  call `/mcp`.
- `MODAL_MCP_SIGNING_KEYS` or `MODAL_MCP_SIGNING_KEY_FILE`: one or more
  `kid:hex` HMAC signing keys for refs, cursors, and approval tokens.
- Modal credentials from either `MODAL_TOKEN_ID` + `MODAL_TOKEN_SECRET`, their
  `_FILE` variants, or an existing `MODAL_CONFIG_PATH`.

Recommended defaults:

- `MODAL_MCP_READ_ONLY=true`
- `MODAL_MCP_ENABLED_TOOLSETS=discovery,apps,containers,logs,volumes,sandboxes`
- `MODAL_MCP_AUDIT_LOG=stdout`
- `MODAL_MCP_HTTP_BIND=127.0.0.1:8765`

Hosted OAuth delegation for Modal tokens is not part of v1. Cloudflare Workers
and other edge runtimes are unsupported targets because the server depends on
the Modal Python SDK, local process hardening, and FastMCP Streamable HTTP
lifespan behavior.

## Security Posture

The server enforces Origin and Host allowlists before MCP handling. Policy
middleware applies rate limits, toolset gates, read-only gates, approval gates
for future mutations, audit events, and output redaction. Known secrets are
redacted after exception formatting and before JSON rendering.

Approval tokens are HMAC signed, session-bound, single-use, and backed by an
approval ledger. In v1, change and expert tools remain disabled by default; the
approval path is present so future mutating tools can keep the same contract.

See [docs/self-hosting.md](docs/self-hosting.md) and
[docs/threat-model.md](docs/threat-model.md).

## Validation

Install the local command runners before using the validation targets:

```bash
brew install just shellcheck actionlint betterleaks semgrep
```

On Linux, install the same tools with your package manager or the upstream
release packages, then run:

```bash
just setup
just pre-commit
just pre-push
just check
just uv-audit  # optional local uv.lock dependency audit
```

Local secret scanning uses Betterleaks through the `just` targets. CI runs
TruffleHog for repository secret scanning.

Live Modal tests are opt-in:

```bash
MODAL_MCP_LIVE=1 MODAL_ENVIRONMENT=dev uv run pytest tests/integration/live -q
```
